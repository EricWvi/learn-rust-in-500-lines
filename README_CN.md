## 这个系列想走第三条路

你大概见过两种 Rust 教程。一种是一上来就 `fn main()` 配 `println!`，用零碎的小例子讲语法，例子之间毫无联系——读完你记得一堆碎片，却不知道怎么把它们拼成一个真正的项目。另一种是直接甩给你一份几万行的开源仓库，说"读源码吧"——可你连入口都找不到。

这个系列想走第三条路：**我们只读一个项目，一个真实、完整、但只有大约 500 行的项目**，从它的第一行读到最后一行。Rust 里那些最让人头疼的概念——所有权、生命周期、`trait`、异步、通道——不是凭空蹦出来吓你的，它们每一个都是这 500 行里**为了解决某个具体问题**才出现的。我们就在那个问题发生的现场，把它们一个一个认出来。

## 「500 行」是怎么来的

先说这个数字怎么来的。整个系列围绕的，是 `code/` 目录下一个叫 `process` 的小 crate——它的作用是"帮你启动并管理子进程"。不算测试，它的核心源码一共 **512 行**：

- `src/lib.rs` —— 7 行，整个 crate 的正门；
- `src/spec.rs` —— 153 行，描述"要启动一个什么样的子进程"；
- `src/traits.rs` —— 53 行，定义对外的接口约定；
- `src/tokio_process.rs` —— 299 行，真正干活的实现。

512，约等于 500，这就是系列名字的由来。

为什么偏偏挑这个体量？因为它卡在一个甜点上：

- **小到能装进脑子。** 500 行你可以从头到尾通读，记得住每个文件负责什么，不会被规模淹没。
- **大到足够真实。** 它不是那种只为教学拼凑的玩具——它有清晰的分层（描述 / 接口 / 实现），有真正棘手的并发问题（要同时等子进程退出、等"杀死"命令、等"销毁"信号），也有为了正确性而做的妥协和注释。换句话说，**它逼着你碰 Rust 的硬骨头**。

一句话：500 行足够小，小到学得完；又足够真，真到学完能用。

## 你甚至可以不打开 IDE

很多教程的潜台词是"先把仓库 clone 下来，用 IDE 打开，跟着源码看"。这个系列不要求你这么做。

每一篇文章里，**相关的代码会原样贴在正文当中，并逐行解释**。你完全可以躺在沙发上用手机读、用平板读、打印成纸读，从头到尾不碰电脑、不装工具、不 clone 任何东西，也能跟上每一行在发生什么。

当然，如果你想动手——这些代码是真的，能编译、能跑。仓库里的 `code/` 就是那个 crate 本体，`cargo test` 一敲，测试全绿。读到一个地方手痒了，随时可以打开 IDE 改两行，看看会发生什么。但这是**加分项**，不是**入场券**。

## 由浅入深：这条路怎么走

这个系列的顺序，是刻意按"从外到内、从易到难"排的。大致路线是这样：

1. **先认骨架**：一个 Rust 项目长什么样？什么是 `crate`、`lib.rs`、`mod`、`Cargo.toml`？我们先把地图摊开。
2. **再读数据**：`ProcessSpec` 那份"清单"是怎么搭起来的？由此认识 `struct`、字段、方法，以及 Rust 最特别的**所有权（ownership）**。
3. **再读约定**：`ProcessSpawner` 和 `ManagedProcess` 这两个接口是怎么定义的？由此认识 `trait`——Rust 里表达"共同能力"的方式。
4. **最后读实现**：`tokio_process.rs` 那 299 行里，怎么用异步和通道，把一个子进程的整个生命周期管起来？这是最难的一段，但走到这里时，你已经攒够了所有前置概念。

你会发现，每一篇都只在前一篇的地基上加一层。**没有一篇会用到它还没讲过的东西。** 这是这个系列给你的承诺：只要你按顺序读，就不会在某一天突然撞上一堵没解释过的墙。

## 怎么读最有效

- **按顺序读。** 概念是一层层铺垫的，跳读容易在第三层踩空。
- **术语只定义一次。** 一个词第一次出现时，我会用日常类比把它讲透，并记进仓库里的术语表 `terminology.md`；之后再遇到它，我不会重复定义，而是直接用。所以如果某个词记不清了，去翻那张表。
- **代码是主角。** 每篇文章都从一段真实代码出发，最后回到对这段代码的理解。读的时候，多问自己一句"这一行是为了解决什么问题，才写在这里的"。
